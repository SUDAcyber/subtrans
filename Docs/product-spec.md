# 字幕锻造产品构思

## 目标

做一个面向超长字幕文件的 macOS 翻译工作台。它不是聊天工具，而是一个把字幕结构锁死、把模型限制在“只翻译文本”范围内的批处理系统。

## 第一版能力

- 导入 `.srt` 文件，支持 UTF-8、UTF-16、GB18030 等常见编码。
- 本地解析字幕序号、时间轴和文本。
- 目标语言可指定，默认简体中文。
- 默认 AIHubMix：`https://aihubmix.com/v1`，模型默认 `gpt-5.5`。
- 支持任意 OpenAI 兼容服务：可改接口地址、密钥、模型名。
- 支持聊天补全和 Responses 两种接口模式。
- 支持推理深度、输出长度、分段大小、上下文重叠、失败重试。
- 密钥存入 macOS Keychain。
- 后台记忆库：保存人名、泰语名和术语的固定译法，并在每批请求里注入模型提示。
- 译文查找替换：搜索当前文档译文，支持替换一个或全部匹配项。
- 翻译完成后自动在原字幕文件夹下生成一个带目标语言的新版本。
- 历史记录持久保存，支持移到回收箱、恢复、手动永久删除和 15 天自动清理。
- 对疑似人名但未在记忆库确认的字幕行进行高亮，提醒人工复核。
- 翻译后本地重建 SRT，保留原序号和原时间轴。
- 导出 UTF-8 SRT。

## 长字幕稳定策略

### 不把完整 SRT 交给模型

完整 SRT 一次性进入模型容易出现：

- 时间轴被改写
- 序号错位
- 大段漏译
- 模型自行合并或拆分字幕
- 输出被截断

字幕锻造的策略是：模型永远不负责输出最终 SRT，只负责返回指定 cue id 的译文。

### 本地解析

导入后先解析为：

```text
sequence
startTime
endTime
sourceText
translatedText
```

导出时使用原始 `sequence/startTime/endTime` 重建，最大限度避免字幕错位。

### 分批请求

默认每批 60 条、约 6000 源字符，并带前后各 6 条上下文。上下文只用于理解，不要求模型返回。

请求结构：

```json
{
  "targetLanguage": "简体中文",
  "contextBefore": [],
  "focusedCues": [],
  "contextAfter": []
}
```

返回结构：

```json
{
  "translations": [
    { "id": 1, "text": "译文" }
  ]
}
```

### 校验

每批返回后本地检查：

- 返回 id 是否完整
- 是否出现不属于该批次的 id
- 是否有空译文
- 是否存在重复序号
- 是否需要去掉中文标点

失败批次会按设置自动重试。

## UI 区域

### 左侧边栏

- 文件列表
- cue 数量和文件大小
- 当前进度和校验状态
- 历史记录和回收箱

### 中央工作区

- 顶部任务状态
- 开始任务、停止任务、生成字幕和另存为
- 译文查找替换工具条
- 批次进度
- 原文和译文并排预览
- 序号和时间轴单独固定显示
- 超长文件默认只渲染前 800 条，避免界面卡顿，导出仍包含全部 cue

### 右侧检查器

- 接口名称和接口地址
- 密钥
- 模型名
- 聊天补全 / Responses
- 目标语言
- 推理深度 / 输出长度
- 每批条数、字符上限、上下文重叠、失败重试
- 译文查找替换
- 后台记忆库
- 翻译指令
- 校验统计

## 后续版本建议

### v0.2 批量队列

- 一次导入多个 SRT
- 串行或并发队列
- 每个文件独立目标语言和模型
- 失败文件一键重跑

### v0.3 术语表

- 人名地名固定译法
- 项目级 glossary
- 每次请求自动注入相关术语

### v0.4 质量检查

- 译文长度过长提醒
- 标点残留检查
- 空译文检查
- 疑似未翻译英文残留检查

### v0.5 成本与速度

- 估算 token 和成本
- 记录每批耗时
- 支持低价模型初译 高质量模型复核

- Xcode 工程和签名配置
- App icon
- Sparkle 自动更新
